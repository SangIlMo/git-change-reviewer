# Research: Checks API Annotation Output Channel

**Date**: 2026-04-03
**Phase**: 0 — Pre-implementation research

---

## 1. GitHub Checks API for Posting Annotations

### Decision

Use `POST /repos/{owner}/{repo}/check-runs` to create a check run, then `PATCH /repos/{owner}/{repo}/check-runs/{check_run_id}` to append annotations in batches of 50.

### Rationale

The Checks API annotations appear inline in the Files Changed tab (diamond icon) and in the Checks tab, without creating entries in the Conversation tab. This directly solves the user's problem.

### Key API Details

- **Create**: `POST /repos/{owner}/{repo}/check-runs` — accepts `name`, `head_sha`, `status`, `conclusion`, `output` (with `title`, `summary`, `annotations[]`)
- **Update**: `PATCH /repos/{owner}/{repo}/check-runs/{check_run_id}` — same schema, annotations are **appended** (not replaced)
- **Annotation schema**: `path` (string), `start_line` (int), `end_line` (int), `annotation_level` (notice|warning|failure), `message` (string), `title` (string, optional), `raw_details` (string, optional)
- **Limit**: 50 annotations per API call (both create and update)
- **Conclusion values**: `success`, `failure`, `neutral`, `cancelled`, `skipped`, `timed_out`, `action_required`

### Alternatives Considered

- **PR review comments (current)**: Works everywhere but clutters Conversation tab
- **Commit status API**: Only provides pass/fail status per commit, no inline annotations
- **GitHub Actions workflow annotations**: `::notice` workflow commands — limited to Actions, no API control

## 2. Token Permission Requirements

### Decision

Checks API requires GitHub App installation tokens. All other token types fall back to review comments.

### Rationale

Research confirmed that only GitHub App tokens have write access to the Checks API:

| Token Type | Checks Write | Source |
|---|---|---|
| GitHub App installation token | Yes | GitHub Docs |
| Fine-grained PAT | No | GitHub Docs — Checks permission not available |
| Classic PAT (`repo` scope) | No | OAuth apps can only read checks |
| `gh` CLI OAuth token | No | OAuth apps can only read checks |
| Actions GITHUB_TOKEN (default) | Read only | Unless generated by a GitHub App |

### Implications for Architecture

- **CI (GitHub Actions)**: Users must set up a GitHub App and pass its token to the action. Alternatively, use `actions/create-github-app-token` to generate an installation token.
- **Local (Claude Code skill)**: Will always fall back to review comments. The `--output-mode=checks` flag is effectively CI-only.
- **Fallback detection**: Attempt check run creation, catch 403/resource not accessible error, then fall back.

### Alternatives Considered

- **Require GitHub App for all usage**: Too restrictive, breaks local workflow
- **Only support review comments**: Doesn't solve the Conversation clutter problem
- **Use `actions/github-script`**: Viable in CI but doesn't help local usage

## 3. Annotation Batching Strategy

### Decision

Create the check run with `status: in_progress`, batch annotations in groups of 50 via sequential PATCH calls, then finalize with `status: completed` and `conclusion: neutral`.

### Rationale

- The 50-annotation limit requires batching for larger PRs
- Using `in_progress` status during annotation posting prevents premature completion
- `neutral` conclusion is appropriate for informational annotations (not pass/fail)
- Sequential PATCH calls append annotations without replacing previous batches

### Flow

1. `POST check-runs` with `status: in_progress`, first 50 annotations
2. For each remaining batch: `PATCH check-runs/{id}` with next 50 annotations
3. Final `PATCH` with `status: completed`, `conclusion: neutral`, summary text

## 4. Mapping Annotation Format to Checks API

### Decision

Map the existing annotation body format to Checks API annotation fields:
- `message` → full annotation body text (markdown)
- `annotation_level` → `notice` (for all annotations)
- `title` → short label based on source type: "Intent (Context)", "Intent (Inferred)", "Intent (Conflict)"
- `path` → file path from change block
- `start_line` / `end_line` → line range from change block

### Rationale

The `title` field provides a quick visual label in the Checks tab. The `message` field preserves the full markdown body. Using `notice` level avoids false alarm fatigue.

### Alternatives Considered

- Map confidence to annotation_level (low→warning): Rejected per clarification — all annotations are informational
- Use `raw_details` for additional context: Unnecessary, `message` field is sufficient
